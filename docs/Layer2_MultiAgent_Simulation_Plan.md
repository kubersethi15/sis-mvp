## 2. What MiroFish Teaches Us

MiroFish spawns thousands of agents to simulate social dynamics. We don't need thousands — 
we need 5-8 well-crafted workplace characters. But the patterns are directly applicable:

### Pattern 1: Knowledge Graph → Agent Personas (ADOPT)
MiroFish uses GraphRAG to extract entities and relationships from seed documents, then 
generates agent personas grounded in that graph. 

**For Kaya:** The seed documents are the JD + employer profile + team context. Instead of 
Neo4j/GraphRAG (too much infrastructure), we use Claude to generate a structured JSON 
"workplace graph" — characters, relationships, tensions, and dynamics derived from the 
actual job context. A Cebuana Lhuillier branch simulation feels like Cebuana Lhuillier, 
not a generic office.

### Pattern 2: Agent Memory (ADOPT, SIMPLIFIED)
MiroFish uses Zep Cloud for persistent agent memory across simulation rounds.

**For Kaya:** We use Supabase (already connected) to store character state per simulation. 
Each character remembers what the candidate said/did in earlier rounds. If the candidate 
resolved a conflict with a customer in round 2, the manager character references that in 
round 4. This creates progressive assessment — skills build on each other.

### Pattern 3: Multi-Agent Interaction (ADOPT)
MiroFish agents interact with each other, not just with the user. Emergent behavior comes 
from agent-to-agent dynamics.

**For Kaya:** Characters don't just talk to the candidate — they talk to each other in 
front of the candidate. A colleague asks the manager for help while the candidate is 
dealing with a customer. The candidate has to navigate competing demands. This is the 
"unscripted" assessment that no interview question can test.

### Pattern 4: ReportAgent (ADOPT)
MiroFish has a separate ReportAgent that observes all interactions and produces a 
structured prediction report.

**For Kaya:** An Observation Agent watches the entire simulation silently, extracts 
behavioral evidence using our existing STAR+E+R framework, and produces the Gate 3 
readiness assessment. This is essentially our extraction pipeline Stage 2-3 applied to 
simulation transcripts.

### Pattern 5: Swarm Scale (SKIP)
MiroFish spawns thousands of agents. We don't need that. 5-8 characters in a focused 
workplace scenario produces richer assessment data than hundreds of shallow interactions.

### Pattern 6: Social Media Framing (SKIP)
MiroFish simulates Twitter/Reddit. We simulate a workplace — branch office, team meeting, 
customer service counter, group chat.

## 3. The Candidate Experience

### How It Feels (UX Flow)

The candidate has already completed Layer 1 (LEEE conversation with Aya). Their skills 
profile exists. Now:

**Step 1 — Briefing (30 seconds)**
"You're about to enter a simulated workplace scenario. You'll be placed in a real situation 
at [Cebuana Lhuillier / employer name] and interact with colleagues and customers. There 
are no right or wrong answers — just respond naturally, the way you would in real life."

**Step 2 — Scenario Entry (immersive)**
The screen transitions from the Aya chat aesthetic to a workplace environment. The 
candidate sees:
- Their role: "You are a new Branch Staff at the Quirino Province branch"
- The setting: "It's Monday morning. You've been here for 2 weeks."
- The cast: 3-5 character avatars with names and roles visible
  (e.g., "Ate Maria — Branch Manager", "Kuya Jun — Senior Teller", "Lola Santos — Customer")

**Step 3 — Multi-Character Conversation (10-15 minutes)**
The simulation runs in rounds. Each round:
1. Something happens (a customer arrives, a colleague asks for help, the manager gives 
   an instruction, two characters disagree)
2. The candidate responds via text (or voice in voice mode)
3. Characters react — to the candidate AND to each other
4. The situation evolves based on what the candidate did

The scenario is NOT scripted turn-by-turn. The Orchestrator Agent decides what happens 
next based on: what evidence has been gathered, what skills gaps remain, and what would 
create the most revealing situation for this specific candidate.

**Step 4 — Debrief (1 minute)**
Aya returns: "That was great. Let me look at what your actions revealed..." 
→ Gate 3 readiness assessment is generated.

### Key UX Principle
It should feel like a day at work, not a test. The candidate makes decisions, helps 
people, navigates tensions. The assessment happens invisibly.

### Component Breakdown

**A. Workplace Graph Generator** (new)
- Input: JD + employer profile + team context
- Output: JSON structure with characters, relationships, setting, initial tensions
- Uses Claude to generate — not GraphRAG/Neo4j (too much infrastructure for now)
- Example output:
```json
{
  "setting": "Cebuana Lhuillier branch, Quirino Province, Monday morning",
  "candidate_role": "New Branch Staff, 2 weeks in",
  "characters": [
    {
      "name": "Ate Maria",
      "role": "Branch Manager",
      "personality": "Warm but stressed. Juggling corporate targets and team morale.",
      "sp_dynamic": "Expects pakikiramdam — won't ask for help directly",
      "goal": "Needs the candidate to handle the floor so she can do reports",
      "relationship_to_candidate": "Supportive but watching closely"
    },
    {
      "name": "Kuya Jun",
      "role": "Senior Teller",
      "personality": "Experienced, slightly territorial. Helpful if approached with respect.",
      "sp_dynamic": "Utang na loob — helped the candidate on day 1, expects reciprocity",
      "goal": "Wants to leave early for a family matter but won't ask directly",
      "relationship_to_candidate": "Mentor figure, testing if candidate notices his stress"
    },
    {
      "name": "Lola Santos",
      "role": "Regular customer, elderly",
      "personality": "Confused by the new system. Needs patience and clear explanation.",
      "sp_dynamic": "Hiya — embarrassed to admit she doesn't understand",
      "goal": "Wants to send money to her OFW daughter",
      "relationship_to_candidate": "Stranger, but expects warmth and respect"
    }
  ],
  "initial_event": "The branch just opened. Ate Maria is in the back room. Kuya Jun 
   looks distracted. Lola Santos walks in looking confused.",
  "assessment_targets": ["Customer Service", "Communication", "Problem Solving", 
   "Emotional Intelligence", "Adaptability"]
}
```

**B. Simulation Orchestrator** (new — the "Brain")
- Manages the simulation loop
- Each round: decides what happens, which character acts, what pressure to apply
- Adaptive: if the candidate has shown strong customer service, introduce a harder 
  challenge. If they haven't shown problem-solving, create a situation that requires it.
- Uses the same adaptive logic as Enhancement #1 (Adaptive Conversation Intelligence) — 
  real-time evidence quality assessment driving what happens next.
- Decides when to end: when enough evidence has been gathered across target skills, 
  OR after max 8-10 rounds.

**C. Character Agent** (extend existing)
- Currently: single character per simulation, 3-round limit
- New: each character is an independent agent with its own system prompt
- The Orchestrator calls `execute(character_name, situation_update) → response`
- Characters can react to the candidate AND to each other
- Character state (memory) persists across rounds in the simulation session

**D. Observation Agent** (new — the "ReportAgent")
- Watches the full simulation transcript silently
- After each round: extracts behavioral evidence using STAR+E+R framework
- Maps evidence to PSF skills with confidence scores
- At simulation end: produces the Gate 3 readiness assessment:
  - Readiness index (0-100)
  - Skills demonstrated in simulation (with evidence quotes)
  - Skills NOT demonstrated (gaps)
  - Success conditions for the role
  - Support needs identified
  - AI recommendation for the Final Approver

**E. Simulation UI** (new page: `/simulation`)
- Extends the LEEEChat aesthetic but with a workplace feel
- Shows character avatars and names
- Multi-party conversation view (not just 1:1)
- Round indicator showing simulation progress
- Transition animation from Aya chat → workplace simulation → back to Aya for debrief

## 5. Integration with Existing Pipeline

```
Layer 1 (LEEE)          →  Layer 2 (Simulation)    →  Gate 3 (Assessment)
                            
Aya conversation             Workplace scenario          Readiness report
  ↓                           ↓                           ↓
Skills extraction            Behavioral observation      Final Approver reviews
  ↓                           ↓                           ↓
Layer 2 seeds generated      Evidence extracted           Select / Decline
  ↓                           ↓
Evidence gaps identified     Gaps targeted by Orchestrator
```

**Data flow:**
1. LEEE extraction produces `layer2_seeds` and `skills_profile` with evidence gaps
2. Workplace Graph Generator uses seeds + vacancy + employer data to create scenario
3. Simulation Orchestrator targets the evidence gaps specifically
4. Observation Agent produces evidence in the same format as LEEE extraction
5. Gate 3 combines LEEE evidence + simulation evidence for readiness assessment
6. Final Approver sees the complete picture: self-reported + observed

**What we reuse:**
- Extraction pipeline Stages 2-3 (evidence extraction + skill mapping) → for simulation evidence
- The `[SIMULATION EVIDENCE]` tag handling already in extraction-pipeline.ts
- Gate 3 API routes (extend, don't rebuild)
- MicroSimulation.tsx patterns (extend to multi-character)
- ScenarioCard.tsx for scenario briefing
- The entire anti-gaming framework (characters can probe inconsistencies between what the 
  candidate told Aya and how they behave in simulation)

## 6. Anti-Gaming in Simulations

This is where Layer 2 becomes powerful for triangulation:

1. **Story-simulation consistency:** If the candidate told Aya they're great at customer 
   service, but freezes when Lola Santos walks in confused, that's a flag.
2. **Behavioral vs. declared:** The simulation tests DOING, not SAYING. Can they actually 
   de-escalate? Actually prioritize? Actually notice someone's distress?
3. **Unscripted responses:** Unlike an interview where candidates prepare answers, the 
   simulation creates novel situations. Rehearsed responses don't work when Kuya Jun 
   suddenly asks for help while Lola Santos is mid-transaction.
4. **SP dynamics as assessment:** Filipino social dynamics (hiya, pakikiramdam, utang na 
   loob) create natural assessment pressure that candidates from the culture navigate 
   instinctively — or don't. This is culturally fair assessment, not cultural bias.

## 8. What This Changes About Kaya's Position

**Before:** Kaya extracts skills from stories (Layer 1) and has placeholder stubs for 
simulation and interview (Gate 3). The three-layer triangulation promised in the 
architecture exists only in design.

**After:** Kaya has genuine three-layer assessment:
- Layer 1: What the candidate SAYS about their skills (LEEE conversation)
- Layer 2: What the candidate DOES when placed in a workplace (multi-agent simulation)
- Layer 3: What others observe about the candidate (psychologist validation + future peer review)

**Competitive position:** Workera verifies skills through adaptive tests. Kaya verifies 
skills through workplace simulation. Tests measure knowledge. Simulations measure behavior. 
For PWDs who may not test well but perform brilliantly in real situations, this is the 
more equitable assessment.

**The line for Ryan:** "We don't just ask people about their skills or test them on 
knowledge. We put them in a simulated workplace and watch what they actually do. That's 
the assessment method that works for the people traditional systems overlook."

## 10. Decision Points for Ryan

1. **Scenario scope for Phase 1:** Build for Cebuana Lhuillier branch staff specifically, 
   or generic customer service? Recommend: Cebuana-specific — it's the pilot, and 
   specificity makes the demo more compelling.

2. **Number of characters:** 3 characters (simpler, faster to build, good enough for 
   assessment) or 5-8 (richer dynamics, more realistic, more evidence)? 
   Recommend: Start with 3, expand to 5.

3. **Session length:** 6-8 rounds (~10 minutes) or 10-15 rounds (~20 minutes)? 
   Recommend: 8 rounds max. Shorter is better for PWDs with fatigue concerns.

4. **Voice mode priority:** Include voice in Phase 1 or defer to Phase 3? 
   Recommend: Defer. Text-first, same as LEEE. Voice is Phase 3.

5. **Character-to-character interaction:** Should characters talk to each other in front 
   of the candidate (richer, more complex) or only to the candidate (simpler)? 
   Recommend: Yes — this is what makes it feel like a real workplace, not an interview.
